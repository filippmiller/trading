## 1. Observability / Alerts (Discord webhook) — CRITICAL, до понедельника

**Цель:** узнавать о падениях cron в реальном времени, а не при ручном осмотре.

**Компоненты:**
- Новая переменная окружения `DISCORD_WEBHOOK_URL` в `docker-compose.surveillance.yml`
- Новая функция `postAlert(level, message)` в `scripts/surveillance-cron.ts`: POST JSON в Discord webhook, fetchWithTimeout(5s), тихо fail если webhook пуст
- Триггеры:
  - `surveillance_logs.status='FAILED'` при любом завершении sync/enroll/scan
  - Exception в scheduled callback (обёртка вокруг `try { await job() } catch { postAlert(...) }`)
  - Startup complete notification (one-shot: "Surveillance cron started, next run at X")
  - Daily summary (вечерний catchup): "Today: N enrollments, M signals, $X realized P&L"
- Опционально: `/api/alerts/test` endpoint для ручной проверки webhook

**Риски:**
- Discord webhook rate limit — throttle до 1 msg/5s, последующие дропать
- Бесконечный loop если alert сам падает — обернуть `postAlert` в try/catch который только console.log
- Secret в webhook URL — не коммитить, через .env

**Effort:** 30-45 мин  
**Dependencies:** нет  
**Definition of done:** тестовый webhook срабатывает, failed job в prod → сообщение в Discord

## 2. Database backups — CRITICAL, до понедельника

**Цель:** защита месяцев работы от случайного `docker volume rm` или corruption.

**Компоненты:**
- Новый скрипт `scripts/backup-trading-db.sh` на VPS в `/opt/trading-surveillance/scripts/`
  - `docker exec <mysql container> mysqldump -uroot -p$MYSQL_ROOT_PASSWORD trading | gzip > /var/backups/trading/trading-$(date).sql.gz`
  - Retention: 14 дней, cron `find ... -mtime +14 -delete`
- Systemd timer `/etc/systemd/system/trading-backup.{service,timer}`
  - Daily 03:15 UTC (= 22:15 or 23:15 ET, после всех jobs)
  - Alert на Discord если backup failed (`ExecStartPost=` или wrapper)
- Опционально: upload в S3 / remote — нужен AWS key или Hetzner Object Storage
- Тестирование восстановления — один раз вручную: `zcat latest.sql.gz | mysql -uroot ...` в scratch DB

**Риски:**
- mysqldump под нагрузкой может блокировать — использовать `--single-transaction --quick`
- Disk fill — 14 дней × 50MB ≈ 700MB, мало для VPS но мониторить
- Secret MYSQL_ROOT_PASSWORD в script — читать из `/etc/trading-backup.env` mode 600

**Effort:** 1 час  
**Dependencies:** нет  
**Definition of done:** первый бэкап в `/var/backups/trading/`, systemctl status зелёный, restore test прошёл

**Компоненты:**
- Новая страница `/research/compare` или режим "compare mode" на `/research`

- Visualization:

  - Diff-таблица метрик: A vs B | Δ (highlight where A > B)

  - Overlap histogram (где сделки пересекаются)

**Риски:**
- Много состояния на странице — держать в useReducer или разбить на компоненты
- SVG рендер с 2 curves: масштабирование по общему min/max

**Effort:** 2 часа  
**Dependencies:** none (opt Phase 3 из PR #5 нужен для loading saved, но уже есть)  
**Definition of done:** кнопка "Compare mode" добавляет второй форм-блок, Run runs обе, diff таблица показывает outperformer

## 4. Live status dashboard

**Цель:** одна страница которая показывает «что сейчас происходит / работает ли всё».

**Компоненты:**
- Новая страница `/status` или обновить `/` главную
- Секции:
  - **Cron health:** last successful run, next scheduled (countdown), 5 последних entries из surveillance_logs
  - **Open positions:** count, total invested, total open equity, по directions LONG/SHORT
  - **Cash:** сумма cash across всех paper_accounts, initial vs current diff
  - **Today's activity:** enrollments, signals fired, exits triggered
  - **Data pipeline:** сколько reversal_entries имеют d1_close заполненный, stale count
- API endpoint `/api/status` — один запрос, все агрегаты, cache 30s
- Auto-refresh every 30s на странице

**Риски:**
- Много агрегатных queries — держать их быстрыми (COUNT(*) на indexed columns)
- Time-zone display — всегда ET в UI

**Effort:** 2-3 часа  
**Dependencies:** нет  
**Definition of done:** `/status` показывает current state, можно на глаз увидеть что прошёл 09:50 tick, сколько сделок открыто

## 5. Grid search (2D parameter sweep)

**Цель:** 1D sweep находит локальный оптимум; 2D видит глобальный.

**Компоненты:**
- Расширить `POST /api/research/sweep` → `POST /api/research/grid`
  - Принимает `dimX` и `dimY` вместо одного `dim`
  - Для каждой пары (x, y) зовёт `runScenario`, возвращает matrix
  - Bounded: 2..20 шагов per dim, max 2..100 total points, timeout 60s
- UI: новая секция на `/research` "Grid Search" с select X/Y + from/to/step для каждого
- Visualization: heatmap
  - SVG grid, клетки цветом (green → red gradient по totalPnlUsd)
  - Hover → tooltip с full metrics для этой клетки
  - Best cell подсвечен 🏆
- Опционально: click на клетку → применить те параметры в основную форму

**Риски:**
- 20×20 = 400 scenarios = долго, нужен server-side timeout + UI loading bar
- Mem — каждый scenario возвращает full trades[]; для grid возвращать только summary
- Цветовая шкала — решить что "baseline" (0, или медиана по grid)

**Effort:** 2 часа  
**Dependencies:** существующий sweep endpoint (есть после PR #5)  
**Definition of done:** выбор `exit.holdDays × trade.leverage`, Run Grid → 5×5 heatmap показывает где оптимум
