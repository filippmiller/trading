## Context

User asked for a comprehensive critic review of the `C:\dev\trading` project — a mean reversion hypothesis testing platform. The core hypothesis: shorting stocks that gained 2-3 days straight and buying stocks that fell 2-3 days straight should be profitable. The system should record market open/close prices and intraday snapshots for the most active, most losing, and most winning stocks daily.

The review was performed using 5 parallel analysis agents (build correctness, bug scan, security, consistency, edge cases), followed by systematic fixing and verification.

2. **SAR flip cursor overwrite** (`backtest.ts:360-381`): After flip trades, loop counter was unconditionally reset to original stop bar, re-processing already-consumed bars. Fixed with conditional branching.
3. **SQL injection in ensureColumn** (`migrations.ts:91-101`): Table/column names were interpolated into DDL. Added whitelist validation + regex check. Also fixed TOCTOU race by catching duplicate column error (errno 1060) instead of check-then-act.

#### High Fixes
4. **Missing "most active" stocks** (`movers/route.ts`): Added `fetchMovers("most_actives")` as 3rd category from Yahoo Finance screener

6. **CSV parser accepts bad data** (`data.ts:14-31`): Added `isNaN` checks and `close <= 0` guard — malformed Stooq rows now skipped instead of stored as 0/NaN
7. **LIMIT interpolation** (`data.ts:97`): Changed from `pool.query` with string interpolation to `pool.execute` with parameterized `LIMIT ?`
8. **Input validation** (`reversal/route.ts`): Added date format regex, direction allowlist (LONG/SHORT), positive price check

## Problems Encountered

- The SAR flip cursor bug was subtle: the unconditional `i = exit.exitIndex + 1` on line 379 ran after the flip loop's own `i` assignments, silently overwriting them. This could cause trades to be generated from already-processed bars, inflating trade counts and P&L in SAR strategies.
- The `daysHeld` fix has an edge case: if two measurement slots have the same numeric price value (e.g., d1_close and d2_morning are both $50.00), the `===` comparison might match the wrong field. A more robust fix would track the field name alongside the value, but this is unlikely in practice with 6-decimal precision.

## Gotchas & Notes for Future Agents

1. **Three structural gaps remain unresolved** (need user design decisions):
   - No automated price collection — all 9 measurement cells are manual entry. Need a cron/scheduler calling Yahoo Finance at 9:30 AM, 12:00 PM, 3:55 PM ET.
   - Entry price is captured at button-click time, not market close. Should either force manual entry at 3:55 PM or auto-fetch close prices.
   - Regime filter (MA200) logic is inverted for fade strategies — blocks shorting overextended stocks, which is exactly what fade wants to do.

2. **ensureColumn definition parameter** is still string-interpolated in DDL (unavoidable — MySQL doesn't support parameterized DDL type definitions). It MUST remain a hardcoded literal at every call site.

3. **The movers API now returns a `mostActives` field** but the frontend UI (`reversal/page.tsx`) doesn't display it yet. The `MoversData` type on the page needs updating to include `mostActives: Mover[]` and a third selection panel.

4. **`markCompleted` in page.tsx** still reads from stale React state instead of fetching fresh data from DB. This was identified but not fixed because it requires either a GET handler on `/api/reversal/[id]` or ensuring loadCohorts() completes before the Close button is clickable.
