## P0 — Can corrupt trading data / lose money under a real broker

### P0-1. `jobMonitorPositions` has no concurrency guard → double exit / double cash refund

**File:** `scripts/surveillance-cron.ts:707-965`, scheduled at `scripts/surveillance-cron.ts:1364`

**What's wrong:** Every other job has a module-level boolean guard (`syncRunning`, `enrollRunning`, `trendScanRunning`). `jobMonitorPositions` has none. It's scheduled `*/15 9-16 * * 1-5` (line 1364), and each run can take 30-60+ seconds if Yahoo is slow and there are 20+ open positions. Two overlapping runs can:

1. Both `SELECT * FROM paper_signals WHERE status='EXECUTED' AND exit_at IS NULL` → same rows returned
2. Both evaluate exit conditions → both decide to exit
3. Both `UPDATE paper_signals SET status='WIN'/'LOSS', pnl_usd=?...` — the second overwrites but that's OK
4. **Both `UPDATE paper_accounts SET cash = cash + proceeds` → cash is credited twice.** No idempotency — the UPDATE doesn't guard on `status = 'EXECUTED'`.

Same double-credit hits pending order fills (line 927-931) because `paper_orders.status='PENDING'` is checked but the UPDATE to FILLED and the UPDATE to cash are non-atomic.

**Why it matters live:** Your session log shows 5 positions still open at end of day 1. On day 2, a 15-min tick overlap could silently double-refund the exit proceeds for QLYS ($1,535 → $3,071 phantom cash). This is exactly the class of corruption you built paper-trading to catch.

**Suggested fix:**
```ts
let monitorRunning = false;
async function jobMonitorPositions() {
  if (monitorRunning) { log("  SKIP: monitor already running"); return; }
  monitorRunning = true;
  try { ... } finally { monitorRunning = false; }
}
```
Plus gate the exit update on the pre-exit status: `UPDATE paper_signals SET ... WHERE id=? AND status='EXECUTED' AND exit_at IS NULL` and skip the cash credit if `affectedRows === 0`.

### P0-2. Phantom `EXECUTED` signals when cash check fails after signal insert

**File:** `scripts/surveillance-cron.ts:657-680` (`jobExecuteStrategies`) and `scripts/surveillance-cron.ts:1108-1133` (`jobExecuteConfirmationStrategies`)

**What's wrong:** The code inserts the signal with `status='EXECUTED'` **first**, then attempts to decrement cash with a conditional UPDATE (`WHERE id=? AND cash >= ?`). If `affectedRows === 0` (concurrent spend, midday restart, cash raced), the code logs "cash exhausted" and `break`s — but **the signal row is already persisted as EXECUTED**. The next `jobMonitorPositions` sees it, fetches price, eventually exits, and credits proceeds — **cash that was never spent** enters the account.

The in-memory guard `remainingCash < investmentPerTrade` at lines 615 and 1034 does NOT catch this, because on container restart or out-of-band SQL writes (your cleanup-stale-2026-04-16.sql doing refunds) the DB value differs from the cached snapshot loaded at line 593 / 1014.

**Why it matters live:** You just ran a cleanup SQL that refunded cash and canceled signals. Any overlap between the cleanup running and the cron running = phantom money. Also: Sunday container redeploy + pre-market startup catchup fires all three `jobExecute*` functions serially. If the container OOM-kills midway (NODE_OPTIONS caps heap at 768M; cache can exceed this with 500 symbols × 600 bars), the signal write survives but the cash deduction may not.

**Suggested fix:** Reverse the order — deduct cash first (atomically, with the `cash >= ?` guard), then only if `affectedRows > 0` insert the signal. Best done in a MySQL transaction.

```ts
await db.beginTransaction();
const [cashUpdate] = await db.execute(
  "UPDATE paper_accounts SET cash = cash - ? WHERE id = ? AND cash >= ?", ...);
if (cashUpdate.affectedRows === 0) {
  await db.rollback();
  break;
}
await db.execute("INSERT INTO paper_signals ...");
await db.commit();
```

### P0-4. Trading-day loop uses mixed UTC/local date semantics (off-by-one on TZ boundary)

**File:** `scripts/surveillance-cron.ts:478-487`

**What's wrong:**
```ts
for (const row of entries) {
  const entryDate = new Date(row.cohort_date);           // MySQL DATE → UTC midnight
  let tradingDay = 0;
  const obsDate = new Date(entryDate);
  while (tradingDay < 10) {
    obsDate.setDate(obsDate.getDate() + 1);              // LOCAL time arithmetic
    if (obsDate.getDay() === 0 || obsDate.getDay() === 6) continue;  // LOCAL dow
    const dateStr = obsDate.toISOString().split("T")[0]; // UTC date string
```

Three different time semantics mixed together:
- `new Date('2026-04-16')` → UTC 2026-04-16T00:00:00Z (which is 2026-04-15T20:00 ET)
- `obsDate.getDay()` → returns local day-of-week. In ET container with UTC midnight, the local date is actually the PREVIOUS day → getDay returns wrong dow near midnight
- `obsDate.toISOString().split("T")[0]` → gives UTC date, not ET date

Combined with `TZ: America/New_York` in docker-compose (line 42), this creates an off-by-one where Monday entries' d1 observation can be computed as Sunday (weekend-skipped to Monday, i.e., same day), or d1 observation actually lands on the cohort day itself.

**Why it matters live:** Silent corruption of the reversal_entries d-column grid. You might be storing the d2_morning price into d1_morning, which would then drive wrong signal decisions for CONFIRMATION strategies that look at `d1_close`.

**Suggested fix:** Work entirely in ET or entirely in UTC. Using `formatET(obsDate)` everywhere:
```ts
function addDaysET(dateStr: string, days: number): string {
  // Add days using ET calendar semantics — parse as noon ET to dodge DST edges
  const [y, m, d] = dateStr.split("-").map(Number);
  const base = new Date(Date.UTC(y, m - 1, d, 17, 0)); // 17:00 UTC = ~noon ET (accepting ±1hr DST)
  base.setUTCDate(base.getUTCDate() + days);
  return new Intl.DateTimeFormat("en-CA", { timeZone: "America/New_York" }).format(base);
}
```

### P0-5. Auto-close of stale ACTIVE entries orphans live paper_signals

**File:** `scripts/surveillance-cron.ts:466-468`

**What's wrong:**
```ts
await db.execute(
  "UPDATE reversal_entries SET status = 'COMPLETED' WHERE status = 'ACTIVE' AND cohort_date < DATE_SUB(CURRENT_DATE, INTERVAL 14 DAY)"
);
```

Marks a reversal entry COMPLETED after 14 calendar days, but does not touch `paper_signals` still linked to it where `status='EXECUTED' AND exit_at IS NULL`. The only exit_reasons the monitor knows are HARD_STOP, LIQUIDATED, TAKE_PROFIT, TRAIL_STOP, TIME — none fire automatically from the reversal-entry completion. So:

- A signal with `time_exit_days = 10` is fine (TIME exits first at day 10).
- A signal with no time_exit_days (e.g. `Trailing 2%` template has time_exit_days=7, OK) — but custom configs could have no time exit.
- A signal that the monitor can't price-fetch (Yahoo blocks the symbol, symbol delisted) will stay `EXECUTED` forever, cash locked up, and the monitor keeps polling it every 15 minutes.

**Why it matters live:** Delisted tickers (happens with your SPAC/biotech top-losers) will silently lock investment USD in perpetuity. On day 15 the reversal entry is COMPLETED but the paper_signal's `reversal_entry_id` FK still points to it — no cleanup.

**Suggested fix:** After marking reversal_entries COMPLETED, also force-close the linked open signals at their last-known price (or entry price if no prices recorded):
```sql
UPDATE paper_signals ps
JOIN reversal_entries re ON ps.reversal_entry_id = re.id
SET ps.status='FORCE_CLOSED', ps.exit_at=NOW(), ps.exit_reason='COHORT_EXPIRED', ...
WHERE re.status='COMPLETED' AND ps.status='EXECUTED' AND ps.exit_at IS NULL;
```

### P0-6. Monitor's `open_market_value` in `/api/strategies` ignores SHORT direction

**File:** `src/app/api/strategies/route.ts:33-50`

**What's wrong:** The aggregate `open_market_value` computation:
```sql
COALESCE(SUM(
  CASE
    WHEN ps.status = 'EXECUTED' AND ps.exit_at IS NULL THEN GREATEST(
      0,
      ps.investment_usd + (
        ps.investment_usd * GREATEST(
          ((COALESCE(lp.price, ps.entry_price) - ps.entry_price) / NULLIF(ps.entry_price, 0)) * ps.leverage,
          -1
        )
      )
    )
    ELSE 0
  END
), 0) as open_market_value
```

The formula `(live - entry) / entry × leverage` is unsigned — it's the LONG PnL. For a SHORT position, when price goes UP, `(live - entry) / entry` is positive, and this code reports the SHORT as *gaining* market value. The UI's top-3 podium and equity/return % numbers for all strategies with open SHORT positions are wrong.

Compare to `jobMonitorPositions` which correctly flips the sign via `const pnlPct = isShort ? -rawPricePct : rawPricePct;` (line 773). The SQL version never had the same treatment.

**Why it matters live:** Day 1 had 4 SHORT positions gap-stopped. The strategies page would have shown them as *winning* until the exit fired. More importantly, this is the same class of bug as the SHORT-PnL-inversion you already fixed in the monitor — but it still exists in the read-path SQL. Anyone looking at the strategies page during market hours to decide whether to pull a strategy is getting inverted info for SHORT positions.

**Suggested fix:** Join `paper_signals.direction` and multiply raw return by `CASE WHEN ps.direction='SHORT' THEN -1 ELSE 1 END` before multiplying by leverage. Or do the mark-to-market in Node and skip this SQL branch.

### P1-3. Container TZ is ET; `docker/init-db.sql` uses `CURRENT_DATE`/`CURRENT_TIMESTAMP` — MySQL server TZ is not set

**File:** `docker/docker-compose.surveillance.yml:5-7, 36-45`, plus the cron code

**What's wrong:** Cron container has `TZ: America/New_York` (line 42). MySQL container has **no** TZ env — MySQL defaults to SYSTEM which is container's TZ (usually UTC). `DB_CONFIG.timezone: "Z"` in the pool (line 35) means the client tells MySQL "treat our datetimes as UTC." So:

- `CURRENT_DATE` on MySQL evaluates in MySQL server's TZ (UTC by default)
- `CURRENT_TIMESTAMP(6)` ditto — stored as UTC
- Line 467: `WHERE cohort_date < DATE_SUB(CURRENT_DATE, INTERVAL 14 DAY)` — uses UTC's CURRENT_DATE
- Line 604: `DATE(generated_at) = ?` where `?` is `todayET()` — compares UTC date of generated_at to an ET date string

Before 20:00 ET (midnight UTC), `todayET() === DATE(generated_at_UTC)` usually matches. But from 20:00-23:59 ET (which is 00:00-03:59 UTC of the next day), `DATE(generated_at)` returns the NEXT day's date in UTC. So an 18:00 evening catchup job (line 1358) that inserts any row from jobExecuteConfirmationStrategies → `generated_at` is stored as ~22:00 UTC → DATE=today UTC → but `todayET()` already rolled to tomorrow.

Wait — 18:00 ET = 22:00 UTC, so DATE in UTC is still today. Let me re-verify: ET is UTC-5 (or UTC-4 DST). 18:00 ET = 23:00 UTC (DST) or 23:00 UTC. So `DATE(generated_at)` stays today. OK. **But** if you ever add an evening 20:00+ ET job, or the catchup runs at 20:30 ET (post-daylight-savings-fall) = 01:30 UTC next day → **the day-count query at line 603-606 returns 0 for positions just inserted**, because they're tagged as "next day" in UTC but the comparison uses ET's today.

The 16:30 confirmation job (line 1353) at 16:30 ET = 21:30 or 20:30 UTC → still safe (today).

**Why it matters live:** The 18:00 evening catchup line 1358 = 23:00 or 22:00 UTC. Still same UTC day. Safe *today*, but the invariant "ET-today == UTC-today of generated_at" is fragile and will silently break if anyone adds a post-midnight UTC job or the DB server TZ changes. Also: `CURRENT_DATE` at line 467 being UTC means a Monday 20:30 ET restart would auto-close entries using UTC=Tuesday, which differs by 1 day from the ET-based calendar used everywhere else.

**Suggested fix:** Set `TZ: America/New_York` on the mysql service too, OR use `CONVERT_TZ(UTC_TIMESTAMP(), '+00:00', 'America/New_York')` explicitly everywhere, OR pass ET-today as a parameter everywhere instead of relying on MySQL's CURRENT_DATE.

### P1-5. `||` operator treats legitimate 0 as unset on watermark tracking

**File:** `scripts/surveillance-cron.ts:777-780`

**What's wrong:**
```ts
const maxPrice = Math.max(Number(sig.max_price || 0), price);
const minPrice = sig.min_price ? Math.min(Number(sig.min_price), price) : price;
const maxPnl = Math.max(Number(sig.max_pnl_pct || -999), leveragedPnl);
const minPnl = Math.min(Number(sig.min_pnl_pct || 999), leveragedPnl);
```

The `|| -999` and `|| 999` sentinel values replace *any falsy* DB value — including legitimate `0`. Signals are inserted with `max_pnl_pct=0, min_pnl_pct=0` (line 664). On the first monitor tick with price==entry (exactly 0% move), `leveragedPnl = 0`:
- `sig.max_pnl_pct || -999` = `0 || -999` = `-999` → maxPnl = Math.max(-999, 0) = 0 ✓
- `sig.min_pnl_pct || 999` = `0 || 999` = `999` → minPnl = Math.min(999, 0) = 0 ✓

OK first tick is safe. But if subsequent ticks have leveragedPnl stays 0 (volatile whipsaw perfectly back to entry), the same substitution happens, which is still correct. The real issue: when leveragedPnl briefly went negative (say -0.3%) on tick 2, then back to 0 on tick 3:
- After tick 2: max=0, min=-0.3
- Tick 3: `min_pnl_pct || 999` = `-0.3 || 999` = `-0.3` (negative is truthy) → minPnl = Math.min(-0.3, 0) = -0.3 ✓

Hmm, actually `-0.3 || 999` returns -0.3 because -0.3 is truthy. So the operator works for negatives. The ONLY failure mode is `sig.max_pnl_pct === 0` — which gets replaced with -999 — but since `Math.max(-999, anything_reasonable)` returns `anything_reasonable`, max_pnl stays correct unless leveragedPnl is < -999. At 10x leverage, leveragedPnl can only go to -1000 on a -100% crash, which is theoretically possible but capped at -100% in the exit path (line 861: `Math.max(finalPnlPct * leverage, -100)`).

So the practical failure is: `max_pnl_pct` stored as `0` gets replaced with `-999`, but every real `leveragedPnl` exceeds -999, so `Math.max` returns the new value. Hidden effect: on tick 1 after insert, if leveragedPnl < 0 (common for SHORT on gap-up), `Math.max(-999, -3.2) = -3.2` → max_pnl is stored as -3.2 even though it should be max(entry-point=0, -3.2) = 0. **This is the bug** — the best-ever-pnl is the entry point (0%), but we lose that because `0 || -999` erases it.

**Why it matters live:** `best_trade_pct` displayed in the strategies UI (column "Best") reports max_pnl_pct for each trade. For positions that never went positive, it reports the least-negative movement instead of 0. It's cosmetic but it lies — shows "Best: -0.5%" when reality is "Best: 0%" (position never made money).

**Suggested fix:**
```ts
const maxPnl = sig.max_pnl_pct == null ? leveragedPnl : Math.max(Number(sig.max_pnl_pct), leveragedPnl);
const minPnl = sig.min_pnl_pct == null ? leveragedPnl : Math.min(Number(sig.min_pnl_pct), leveragedPnl);
```
