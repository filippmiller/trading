### Phase 2: Initial 3-Day Streak Analysis

Created `scripts/trend-analysis.js` to answer: "Stocks that trended 3 days — did they reverse on day 4?"

**Results from 340 entries with 4+ days of data:**
- 76 stocks had a 3-day streak (same direction d1→d2→d3)
- 57.9% reversed on day 4 (44 stocks)
- 42.1% continued on day 4 (32 stocks)
- DOWN streaks reversed much more: **67.7%** vs UP streaks at 51.1%
- 4-day streaks: 32 stocks; 5-day streaks: only 11 (rare)
- Reversals were slightly larger in magnitude (3.25% vs 2.73%)

**First insight:** DOWN streaks (stocks sliding 3 days) reverse more reliably than UP streaks. Buy-the-dip has a statistical basis.

### Phase 3: Day 2 Direction Change Analysis

User asked: "How many stocks change direction on the second day after the event?"

Created `scripts/trend-analysis-d2.js` analyzing 420 entries.

**Key findings:**
- Overall d1→d2 direction change: 52.9% (coin flip)
- BUT: **Top losers that kept sliding on d1 → bounced on d2 at 61.7%** (58/94)
- Top gainers that kept rising on d1 → pulled back d2 at 54.2% (64/118)
- When direction held on d2, average move was 3.18% vs 2.76% for reversals

**Critical insight discovered:** The 61.7% bounce rate after a 2-day slide was the strongest signal at this point. This led to the hypothesis that confirmation patterns might be even stronger.

### Phase 4: Brainstorming the Research Framework

User said: "Make a list of questions, then run analysis. We need >70% probability scenarios."

Designed a 7-category, 22-question research framework:

1. **Day-by-Day Reversal Timing** — When does reversal happen? (d1-d10)
2. **Event Magnitude as Predictor** — Do bigger drops/spikes reverse more reliably?
3. **Day 1 Pattern as Signal** — Does d1 outcome predict everything after?
4. **Multi-Day Conditional Patterns** — After N consecutive days, what's next?
5. **Entry/Exit Optimization Matrix** — Which buy/sell day combo maximizes win rate?
6. **Combined Filter Stacking** — Stack conditions to find >70% edges
7. **Risk Profile** — Max drawdown before wins, for top scenarios

### Phase 5: The Mega-Analysis (500+ line script)

Built `scripts/mega-analysis.js` — a comprehensive script testing **4,684 scenarios** by combining:
- 2 types (LONG/SHORT)
- 7 magnitude filters (any, >5%, >8%, >10%, >15%, 3-7%, 7-15%)
- 5 day-1 filters (any, favorable, unfavorable, fav >2%, unfav >2%)
- 3 day-2 filters (any, favorable, unfavorable)
- 4 entry points (event close, d1 morning, d1 close, d2 close)
- 10 exit days (d1-d10)

Executed against production database via SSH → docker exec pipe.

**Data coverage:**
- 520 total entries (260 LONG + 260 SHORT)
- Date range: 2026-03-10 to 2026-04-15 (~5 weeks)
- 380 entries with d1-d4 close prices
- 220 entries with full d1-d10 data

## How the Thinking Developed (Analytical Journey)

### Step 1: Naive question → disappointing answer
Started with "do stocks reverse?" — answer was ~50-55%. No edge.

### Step 2: Slicing by direction → first asymmetry
DOWN streaks reverse at 67.7% vs UP at 51.1%. Losers bounce more reliably than gainers fade.

### Step 3: Day 1 as filter → breakthrough
The 61.7% bounce rate for "loser slides d1 → d2 bounce" was the first signal above noise. But still under 70%.

### Step 4: Double confirmation → the real edge
The KEY discovery: filtering by BOTH d1 AND d2 outcome created dramatically higher probabilities:
- If a loser bounces d1 AND d2 → 88-100% chance it's still up on d3
- This is "confirmation momentum" — two days of institutional buying after a big drop

### Step 5: Magnitude stacking → refining the edge
Adding magnitude filters (>5%, >8%, 7-15% drop) concentrated the signal further:
- Loser dropped >5%, bounced >2% on d1, d2 also up → **93.5% win by d3** with +7.7% avg return
- Loser dropped >8%, bounced d1+d2 → **100% win by d3** with +11.7% avg return (n=11)

### Step 6: Risk profiling → surprisingly safe
Max adverse excursion for the best strategies was remarkably low:
- Worst drawdown: -1.07% for the top strategy
- Median drawdown: 0% (most trades never go negative)
- Suggested stop-loss: -2% (would never have been hit in dataset)

### Finding 6: Close Entry Beats Morning Entry

Consistently 3-5% higher win rate when entering at close vs morning. Full day's information improves the signal.

### Finding 7: Optimal Holding Period is d2-d5

Returns peak around d5 for confirmed bounces. By d7-d10 the edge evaporates completely.

## Problems Encountered

1. **Docker container read-only filesystem** — Couldn't `docker cp` the analysis script into the container. Solved by piping via stdin with `node --input-type=module` and `createRequire`.
2. **SSH escaping nightmare** — First attempt to inline the script via SSH heredoc failed with bash parsing errors (backticks, dollar signs, parentheses all conflicting). Solved by scp'ing to `/tmp` on VPS first, then piping into container.
3. **mysql2 module not found** — The container's Node.js couldn't find mysql2 when running via eval. Fixed by using `createRequire('/app/')` to point at the container's node_modules.
4. **Output too large for context** — Mega-analysis produced 122.9KB output (59,516 tokens). Had to read in chunks using offset/limit.

## Gotchas & Notes for Future Agents

1. **The 100% win rates on some scenarios are tautological.** "d1_favorable + entry@event + exit d1" is guaranteed profit because d1_favorable means d1_close > entry_price. The REAL signal is forward-looking: d1+d2 confirmed → d3/d4/d5 continuation.
2. **Sample sizes are small.** 520 entries over 5 weeks. The 20-50 sample scenarios have ~80% confidence intervals of ±5-8%. Need 6+ months for statistical significance.
3. **To run analysis against production DB:** `scp script.js root@89.167.42.128:/tmp/ && ssh root@89.167.42.128 "cat /tmp/script.js | docker exec -i -w /app trading-surveillance-cron node --input-type=module -e 'import { createRequire } from \"module\"; const require = createRequire(\"/app/\"); let d=\"\"; process.stdin.on(\"data\",c=>d+=c); process.stdin.on(\"end\",async()=>{ const fn = new Function(\"require\",\"process\",\"console\",d); fn(require,process,console); })'"` — this is the working pattern.
4. **The MySQL container name is `docker-mysql-1`** (from docker compose), not `trading-surveillance-mysql`.
5. **Market holidays in the cron are hardcoded for 2026-2027** — need annual update.
6. **The "8-12% drop zone" is the sweet spot for losers** — below 8% the signal is weaker, above 12% sample sizes get too small.
