### Phase 2 — API status split (PR #45 `1a626f4`)
- `/api/reversal` already supported `?status=` query param, but the client unconditionally fetched the union (~800KB every time).
- Wired client to derive status from view: `active` → `?status=ACTIVE`, `history` → `?status=COMPLETED`, `matrix` → no status (union).
- Added `fetchedStatus` sentinel so rapid view-toggling doesn't pile up redundant round-trips; treats `ALL` cache as superset of `ACTIVE`/`COMPLETED`.
- Switched `loadData` internals from `Promise.all` to `Promise.allSettled` so one slow arm doesn't tank the other (settings vs reversal).
- Also recovered orphaned `/log` session notes (`2026-04-22-204857.md`) and the 20:48 agent-log wrap entry in a second commit — they'd been lost across a prior branch-switch without commit.

### Phase 3 — Matrix virtualization (PR #46 `acf85ea`)
- Installed `@tanstack/react-virtual ^3.13.24`.
- Flattened cohort-header + ticker rows into `flatRows: FlatRow[]` with `{kind: 'cohort' | 'ticker'}` variants.
- `useVirtualizer({count, estimateSize: 32, overscan: 10})` bound to the outer overflow-auto div via a ref.
- Replaced double-`map` `<tbody>` with single-`map` over visible virtualItems + top/bottom padding `<tr>`s. Each visible row has `data-index` + `ref={rowVirtualizer.measureElement}` for dynamic height refinement.
- Preserved: F1/F2 checkbox state (independent of DOM), scenario-dim + selection-dim opacity, sticky columns (`left-0`, `left-[32px]`), sticky `<thead>`, batch CTA bar.
- Post-deploy verified via one-shot perf-quick script (deleted after):

  - Wall-load: 11,585ms → **4,401ms** (−62%)
  - Longest task: 3,391ms → **812ms** (−76%)

## Technical Decisions

| Decision | Rationale | Alternatives Considered |
|---|---|---|
| Status-split on client, not new endpoint | Server already accepted `?status=`; only the UI needed to start using it | Renaming to `/api/reversal/active`+`/historical` — gratuitous churn |
| `fetchedStatus` cache sentinel instead of per-status cache | Simpler; "if we already have ALL we've got a superset" covers the common case | Keyed Map of responses per status — overkill for MVP |
| Virtualize the existing `<table>` rather than convert to divs | Preserves semantic HTML + sticky-column behaviour + test selectors | Convert to `<div>` grid — bigger blast radius, loses `<thead>` sticky free behaviour |
| Padding `<tr>`s instead of `position: absolute`+transform | Works inside `<tbody>` without CSS fights; react-virtual's recommended table pattern | Transform-based — `<tr>` transforms unreliable across browsers |
| Flat `FlatRow` array instead of nested iteration | Virtualizer needs a single linear index → element mapping | Virtualize per cohort-group — complicated and leaves cohort-headers out of virtualization |
| Overscan 10 rows | Small enough to keep DOM tiny, large enough to mask fast scrolls | 5 (too tight under fast scroll), 30 (negates the DOM savings) |
| Fail open on missing `lastClose` in deviation band (earlier PR #41) | Genuine data gaps shouldn't break batch flow for fresh symbols | Fail closed (too strict, breaks new enrollment flow) |
| Keep prod state clean between probes | User's account is a shared surface; every probe that creates paper trades calls `POST /api/paper/account` reset afterwards | Leave probe state (rejected — clutters UI) |

## Gotchas & Notes for Future Agents

- **Virtualized matrix uses `data-index` + `ref={rowVirtualizer.measureElement}` on each visible <tr>.** Don't refactor `<tr>` into a custom component without forwarding both — the virtualizer needs them to measure dynamic row heights.
- **Sticky columns work via `position: sticky` on `<td>` children** (`left-0`, `left-[32px]`). The virtualizer doesn't interfere as long as the scroll container (`matrixScrollRef`) is the stable overflow-auto parent. If you add a new scroll container inside the matrix, sticky will re-anchor weirdly.
- **Top/bottom `<tr>` padding is how virt empty-space works inside `<tbody>`.** Don't replace them with CSS height on tbody — table layout ignores that.
- **`fetchedStatus === 'ALL'` is treated as a superset** — switching from matrix → active after matrix loaded won't re-fetch, it just filters client-side via existing `grouped`. If you add view-dependent server processing, this assumption breaks.
- **`flatRows` re-computes whenever `grouped` changes.** `grouped` depends on filters (side/ticker/recurrence/streak/sort). So filter changes properly re-flow the virtualizer window. Don't add additional state dependencies to `flatRows` without auditing whether the virtualizer reacts to them.
- **Virtualization does NOT reduce React-render work proportionally** — initial render still mounts all list-overscan rows. The 3.4s → 812ms win is because React only renders ~20 rows instead of ~986, but the wall-load 11.6s → 4.4s savings come partly from the API status-split (PR #45) independently.
- **`/reversal` active-view still renders ~35k DOM nodes.** This is the cards-based layout, separate code path. If user complains about active-view perf, that's another round of virtualization (or lazy-image-card) work, not the same fix.
- **Top-5-gainers flow is still outstanding.** Playwright `.check()` on React-controlled checkbox didn't register. When returning, use direct POST to `/api/paper/batch-order` — more reliable than UI click path.
- **SYSTEM.md at `.claude/SYSTEM.md` documents the canonical topology.** When in doubt, read it before deriving from scratch.
