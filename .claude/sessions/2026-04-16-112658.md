## Context

Extended session continuing from earlier statistical analysis. User wanted to:
1. Build multiple paper trading strategies based on the statistical findings (>70% probability edges)
2. Verify the pipeline was correct end-to-end with two rounds of critical code review
3. Expand the trading universe beyond just Yahoo's top movers — add a trend scanner that detects multi-day consecutive direction streaks in a broader universe (S&P 500-level liquidity)
4. Do multiple rounds of ultra-critical self-review to find subtle bugs that would cause silent failures when the pipeline fires tomorrow

The session uncovered and fixed **7 distinct bugs** across two review rounds plus a third self-review round that found the most severe one (cascade bug from pre-market startup runs).

## What Was Done

### Phase 1: Confirmation Strategy Engine
- Created new `jobExecuteConfirmationStrategies()` cron function that waits for d1/d2 price confirmation before entering trades
- Added new `'CONFIRMATION'` strategy_type distinct from existing `'TRADING'`
- 5 initial strategies created via SQL:
  - Double Confirm Bounce (LONG, >5% drop, d1+d2 favorable) — ~93% win rate expected
  - Big Drop Confirmed (LONG, 8-20% drop, d1+d2 favorable) — ~100% win rate (n=11)
  - Gainer Fade Confirmed (SHORT, >5% rise, d1+d2 favorable) — ~92% win rate
  - Washout Recovery (LONG, >8% drop, d1+d2 UNFAVORABLE) — deep dip buy
  - Momentum Scalp (LONG, >5% drop, d1+d2 favorable, take_profit 3%) — quick exit
- Each strategy funded with $5,000, $100 per trade at 5x leverage
- Cron schedule added: 16:30 ET daily (after close sync)
- UI updated: new "Confirmation only" filter in `/strategies` page

### Phase 2: First Code Review (3 Critical Bugs)
Used `feature-dev:code-reviewer` agent to audit the new code. Found:

**Bug 1:** `jobMonitorPositions()` had LONG-only PnL math. For SHORT positions, exit conditions were inverted — hard stops never fired, trailing stops trailed wrong direction.

**Bug 2:** Confirmation cron scheduled at 16:10 — only 5 min after close sync starts. Risk of race condition if sync takes longer.

**Bug 5:** `paper_signals` table had no `direction` column, making Bug 1 architecturally unfixable.

Fixed all 3:
- Added `direction VARCHAR(8)` to `paper_signals`, backfilled 55 SHORT signals from `reversal_entries.direction`
- Rewrote `jobMonitorPositions()` with direction-aware logic (isShort, inverted pnlPct for shorts, correct trailing stop anchors using minPrice for SHORT vs maxPrice for LONG)
- Moved confirmation cron to 16:30 (25-min buffer)

### Phase 3: Trend Scanner (Universe Expansion)
User asked: "Are we only analyzing top movers? What about stocks sliding/rising for multiple days?"

Identified gap: Yahoo's top 10 gainers/losers screener only captures today's big movers. Stocks in slow multi-day trends (e.g., down 1% for 5 days straight) were invisible.

Implementation:
- Created `scripts/trend-universe.json` with 517 liquid US symbols (S&P 500 + large caps + popular ETFs)
- Updated `docker/Dockerfile.cron` to copy universe file into image
- Added `enrollment_source VARCHAR(16)` column to `reversal_entries` (values: 'MOVERS' or 'TREND')
- Created `jobScanTrends()` function: scans universe with batched parallel Yahoo calls (5 concurrent, 400ms between batches), detects 3+ consecutive same-direction days with ≥3% cumulative move
- DOWN streaks → LONG direction (expect bounce), UP streaks → SHORT direction (expect fade)
- Added `enrollment_source` filter support to both `jobExecuteStrategies()` and `jobExecuteConfirmationStrategies()`
- Added `min_consecutive_days`/`max_consecutive_days` filters to confirmation strategies
- Scheduled trend scan at 16:15 ET (after close sync, before confirmation strategies)
- Created 3 trend-specific confirmation strategies:
  - 3-Day Slide Bounce (LONG, TREND source, 3-4 day DOWN streak, d1+d2 confirmed)
  - 4-Day UP Fade (SHORT, TREND source, 4+ day UP streak, d1+d2 confirmed)
  - Extreme Streak Reversal (any direction, TREND, 5+ day streak, d1 only confirmed)

First scan enrolled 144 stocks with qualifying streaks (MSFT 3d UP, NVDA 6d UP +11.7%, AVGO 6d UP +18.8%, etc.)

### Phase 4: Second Code Review (4 More Bugs)
Second review with `feature-dev:code-reviewer` focusing on the trend scanner integration. Found:

**Bug A:** `fetchDailyBars()` had no timeout — a hanging Yahoo connection would stall entire batch forever since `Promise.all` waits for all 5.

**Bug B:** `jobEnrollMovers()` idempotency counted ALL sources. If TREND scan enrolled for today before 9:45 MOVERS run, the morning enrollment would be silently skipped.

**Bug C:** Trend scan during market hours would capture partial intraday bar as "close" — contaminating streak detection.

**Bug D:** Flat-return days (exactly 0%) classified as "DOWN", incorrectly extending DOWN streaks.

Fixed all 4:
- Added `AbortController` with 8-second timeout to `fetchDailyBars()`
- Updated idempotency query: `WHERE cohort_date = ? AND enrollment_source = 'MOVERS'`
- Added market-hours guard (skip scan during 9:30-16:05 ET)
- Added `if (returns[j] === 0) break;` to streak counter

Initial guard implementation had a flaw (skipped at 00:48 AM mistakenly). Corrected to only skip during active market hours, not pre/post-market.

### Phase 5: Third Ultrathink Self-Review (THE BIG ONE)
User asked for ultrathink critical self-evaluation. I asked myself: "Will the pipeline fire correctly tomorrow?"

**Discovery: Cascade bug causing total Thursday trading day loss.**

At container restart around 00:35 AM ET Thursday 4/16:
1. `jobEnrollMovers` ran at startup WITHOUT time-of-day guard → fetched Yahoo's screener pre-market → enrolled 20 MOVERS entries using Wed 4/15's data but labeled `cohort_date = 4/16`
2. 21 TRADING strategies fired at 00:37 AM → placed **69 paper_signals** against stale entries, deducting cash
3. `jobScanTrends` ran at 00:38 AM → fetched Yahoo daily bars (which only had data through Wed 4/15 close since Thursday market hadn't opened) → enrolled 144 TREND entries with `entry_price = 4/15's close` but `cohort_date = 4/16` (off-by-one-day mismatch)
4. Because idempotency check saw 20 MOVERS entries for 4/16, the REAL 9:45 AM Thursday enrollment would be SILENTLY SKIPPED
5. Downstream: d1_close sync would fill 4/17 as d1 for TREND entries, but entry_price was 4/15 → "d1 return" actually spans 2 trading days, corrupting confirmation strategy logic

**Fixes applied:**
1. `jobScanTrends` now uses `lastBar.date` as `cohort_date` (ensures entry_price ↔ cohort_date always aligned)
2. Added pre-market guard to `jobEnrollMovers`: skips before 9:45 AM ET
3. Updated pre-scan dedup to check last 2 calendar days (handles both today and previous trading day)
4. Enhanced TREND log output to show cohort_date + entry_price + cumulative %
5. Cleanup SQL: refunded cash, marked 69 signals CANCELED with exit_reason='STALE_ENROLLMENT_CLEANUP', deleted 164 stale entries

Verified post-fix: new startup correctly shows `SKIP: too early (01:14:44 ET)` for movers, scan produces `cohort=2026-04-15 entry=$411.22` (self-consistent).

## Technical Decisions

| Decision | Rationale | Alternatives Considered |
|----------|-----------|------------------------|
| Use `lastBar.date` as cohort_date in scan | Guarantees entry_price/date consistency regardless of when scan runs | Use `today` with time-of-day guard to skip pre-market — rejected because brittle |
| Pre-market guard on jobEnrollMovers (9:45 AM cutoff) | Prevents startup runs during pre-market from enrolling with stale screener data | Could gate only startup block; rejected because runtime guard is more defensive |
| Skip scan during 9:30-16:05 ET only (not pre-market) | Pre-market scan uses previous trading day's close which IS final; market-hours would capture partial intraday bar | Skip entire pre-market; rejected because startup after container restart at odd hours should still recover |
| Cancel (not delete) stale signals | Preserve audit trail; mark with exit_reason='STALE_ENROLLMENT_CLEANUP' | Delete — rejected for audit purposes |
| TREND strategies use CONFIRMATION strategy_type | Reuse existing confirmation engine; minimize new code | New strategy_type='TREND' — rejected as unnecessary duplication |
| enrollment_source filter defaults to "ANY" | MOVERS strategies can opportunistically catch TREND entries without explicit filter | Strict isolation via required filter — rejected because limits strategy reuse |
| 5x leverage + $100/trade for new strategies | Smaller positions for risk management on unproven live strategies | $1000/trade like existing TRADING strategies — rejected; want to validate edge on paper first |
| 517-symbol universe inline in JSON | Simple, no external dependency, works in read-only container | Dynamic fetch from Yahoo index constituents — rejected for complexity |

## Problems Encountered

1. **MySQL 8.0 syntax error on `ADD COLUMN IF NOT EXISTS`** — MySQL doesn't support this syntax. Used `SHOW COLUMNS` check first, then plain `ADD COLUMN`.

2. **Docker container read-only filesystem** — Couldn't `docker cp` files into container. Solved by piping scripts via stdin using `docker exec -i` with `node --input-type=module` and `createRequire`.

3. **TypeScript import errors on cron file** — Pre-existing (the file is excluded from tsconfig). Changed `import path from "path"` to `import * as path from "path"` for clarity.

4. **Market-hours guard logic bug** — Initial implementation `curMinutes < 16*60+5` incorrectly skipped at 00:48 AM (post-close next day). Fixed to `>= 9:30 AND < 16:05` — only skip during actual market hours.

5. **The cascade bug (BIGGEST PROBLEM)** — Pre-market startup enrolled 164 stale entries + 69 signals, would have silently blocked legitimate Thursday enrollment. Caught by ultrathink self-review, fixed with two independent guards (pre-market time check + lastBar.date cohort logic).

6. **Duplicate symbol in universe** — BALL appeared twice in trend-universe.json (lines 33 and 39). Smoke test caught it. Removed duplicate on line 39.

7. **Initial test failures** — First smoke test showed 74/75 pass because of BALL dup. After fix: 75/75.

## Gotchas & Notes for Future Agents

1. **`cohort_date` must equal `lastBar.date`, NOT `today`.** This is a load-bearing invariant. If you change the trend scanner or add new enrollment sources, the cohort_date must match the date of the data used to compute entry_price. Mismatching these causes d1/d2 tracking to be off by N days.

2. **`jobEnrollMovers` has a pre-market guard now.** If you need to test movers enrollment manually before 9:45 AM ET, either wait, mock the time, or temporarily comment out the guard. Do NOT enroll pre-market with live Yahoo data — the screener returns stale prior-session data.

3. **Dockerfile has THREE COPY statements for scripts.** If you add a new runtime data file (like universe.json), you must add it to `docker/Dockerfile.cron` explicitly. The container is read-only so runtime file creation is blocked.

4. **`paper_signals.direction` is now required for correct PnL calculation.** Always populate it on INSERT from `reversal_entries.direction`. Missing direction defaults to LONG which will give wrong PnL for SHORT positions.

5. **MOVERS vs TREND semantics differ slightly.** MOVERS entry_price is intraday Yahoo `regularMarketPrice` at 9:45 AM. TREND entry_price is the lastBar close. This means `d1_close > entry_price` has subtly different meaning between the two sources. Not a bug but worth knowing for strategy design.

6. **Confirmation strategies do NOT filter by enrollment_source by default.** If you design a MOVERS-specific strategy, explicitly set `entry.enrollment_source = "MOVERS"` in config_json. Otherwise it will opportunistically match TREND entries too.

7. **To manually run a scan for testing:** `docker logs trading-surveillance-cron --tail 200 | grep -E "Trend|TREND"`. The scan runs at startup (after market close) and at 16:15 ET. Duplicates handled by UNIQUE(cohort_date, symbol) constraint.

8. **Pipeline verification command:** `docker exec -i docker-mysql-1 mysql -u root -p$PASSWORD trading` followed by query. MySQL container name is `docker-mysql-1` (NOT `trading-surveillance-mysql`).

9. **Cash refund SQL pattern:** When canceling stale signals, refund with `UPDATE paper_accounts ... SET cash = cash + investment_usd`. Don't delete signals — change status to CANCELED for audit trail.

10. **The position monitor doesn't need reversal_entry_id.** If you delete a reversal_entries row, paper_signals with that reversal_entry_id will still be monitored correctly (it uses entry_price and live prices directly).
