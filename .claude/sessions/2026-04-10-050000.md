## What Was Done

### Code Review (5 issues found and fixed)

1. **fetchLivePrices concurrency** (`src/lib/paper.ts:28-41`)
   - Was: `Promise.all(unique.map(...))` — fires all Yahoo fetches simultaneously
   - Fix: Process in batches of 5 with sequential await between batches
   - Reason: Unbounded parallel requests risk 429 rate limiting from Yahoo

2. **getDefaultAccount recursion** (`src/lib/paper.ts:52-77`)
   - Was: Recursive call to itself after INSERT — infinite loop if INSERT fails silently
   - Fix: Non-recursive: INSERT, then SELECT, throw if still empty
   - Reason: Defensive against silent INSERT failures (e.g., unique constraint race)

3. **evaluateExit trailing stop high watermark** (`src/lib/strategy-engine.ts:157-163`)
   - Was: `if (pos.current_price >= pos.max_price)` — only updates trail if current > previous max
   - Fix: `Math.max(pos.max_price, pos.current_price)` with monotonic newStop update
   - Reason: If current_price IS the new max, the trailing stop should recalculate from it

4. **computePnL division by zero** (`src/lib/strategy-engine.ts:188-189`)
   - Was: No guard on `entryPrice` before division
   - Fix: Early return `{ pnl_usd: 0, pnl_pct: 0 }` if `entryPrice <= 0`
   - Reason: Defensive against corrupt data with zero/negative entry prices

5. **fillOrder variable shadowing** (`src/lib/paper.ts:227-239`)
   - Was: `const [tradeRows]` in SELL branch shadowed outer scope
   - Fix: Renamed to `const [openTradeRows]`
   - Reason: Variable shadowing can cause confusing bugs in maintenance

## Gotchas & Notes for Future Agents

1. `fetchLivePrices` processes in batches of 5. If you need higher throughput, increase the batch size — but test against Yahoo rate limits first.
2. `getDefaultAccount` throws if the account can't be created. Callers should handle this (all API routes wrap in try/catch, so it surfaces as a 500).
3. The trailing stop in `evaluateExit` is monotonic — it only moves UP, never down. This is by design (trailing stops protect gains, they don't chase losses).
