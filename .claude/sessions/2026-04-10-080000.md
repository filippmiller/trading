## Technical Decisions

| Decision | Rationale | Alternatives Considered |
|----------|-----------|------------------------|
| 2 aggregated SQL queries for strategies API | 48 individual queries took 30s+ over SSH tunnel | Keep individual queries (too slow) |
| Remove ensureSchema from strategies API | It runs 30+ ALTER TABLE statements on every call | Keep it for safety (unacceptable latency) |
| Strategy auto-trade at 9:50 AM (5 min after enrollment) | Ensures today's entries exist before evaluating | Run simultaneously (race condition) |
| Also run strategies at startup | Catches up if container restarted after 9:50 AM | Only scheduled (misses signals on restart) |
| Batch Yahoo fetches at 5 concurrent | Prevents 429 rate limiting | Unlimited parallel (risks blocking) |
| Non-recursive getDefaultAccount | Prevents infinite loop if INSERT fails silently | Retry with backoff (overengineered) |

## Gotchas & Notes for Future Agents

1. **ensureSchema is NOT called in /api/strategies** — intentionally removed because it takes 30s+ over SSH tunnel. Tables are created by other routes or migration scripts.
2. **jobExecuteStrategies runs at 9:50 AM** — 5 minutes after enrollment. If enrollment is delayed, strategies might miss today's entries. They'll catch up on next container restart.
3. **Duplicate signal prevention** uses `(strategy_id, reversal_entry_id)` composite check, not a unique key. If the check races, duplicates are possible but harmless (same position opened twice).
4. **The position monitor** cron `*/15 9-16 * * 1-5` fires during hours 9-16, which includes pre-market (9:00-9:29). Yahoo returns last close price before market opens — this is fine for watermark tracking but exit triggers shouldn't fire on stale pre-market data. Currently no guard for this.
5. **Analysis Only strategies** are skipped by jobExecuteStrategies (type = 'ANALYSIS', not 'TRADING'). They need separate handling if we want to track what they WOULD have done without placing trades.
