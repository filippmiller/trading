## Gotchas & Notes for Future Agents

1. **Lock order is non-negotiable**: `paper_accounts → paper_orders → paper_trades → paper_equity_snapshots`. Every `FOR UPDATE` or mutating write must follow this order. Documented in `src/lib/paper-fill.ts` header. Violating it creates deadlock cycles under concurrent fill/reserve/cancel flows. W1 codex caught this because round-1 had `fillOrderCore` locking orders→accounts while `reserveCashForOrder` did accounts→orders.

2. **Status-guarded UPDATEs + `affectedRows` check are mandatory**: Every `UPDATE paper_orders SET status='...'` needs `WHERE ... AND status='<expected>'`. Every `UPDATE paper_trades SET status='CLOSED'` needs `WHERE ... AND status='OPEN'`. And every such UPDATE must check `affectedRows === 1` — missing check = race-lost state writes silently committed.

3. **Cash/margin mutations always use `AND <col> >= ?` guards**: `UPDATE paper_accounts SET cash = cash - ? WHERE id = ? AND cash >= ?`. Same for `reserved_cash`, `reserved_short_margin`. If `affectedRows !== 1`, rollback.

4. **Codex operating rule — per-file scalpel only**: `codex exec --skip-git-repo-check "<prompt>"` from Bash with `timeout=240000` and `run_in_background: true`. ONE file in the prompt, tight focused questions (5-10 items max), explicit "max N bullets" output cap. Multi-file / full-diff prompts hang indefinitely. Parse output with `awk '/^codex$/,/^tokens used$/'` or just `cat` if the prompt is focused enough that there's no source dump.

5. **Kimi CLI is NOT a codex replacement** — interactive TUI or ACP server only. If codex hangs, fall back to direct OpenAI API curl or a Claude `code-reviewer` subagent. Do not try `kimi -p`.

6. **Smoke tests can lie when derived from implementation**: W1 round-1 had a tautological invariant check (`x === x`). W3 round-1 encoded the SHORT cover arithmetic bug as expected value — 65/65 passed but the math was wrong by $1000/short. When writing smoke tests for financial code, **hand-compute expected values from business rules**, don't derive them from what the code produces.

7. **Two fill paths must call the same module**: `src/lib/paper-fill.ts` is the ONE implementation of `fillOrder`. `scripts/surveillance-cron.ts` (cron) and `src/lib/paper.ts:fillPendingOrders` (UI) both delegate to it. Same for `src/lib/paper-exits.ts` (W3): signal-monitor AND `monitorPaperTrades` both delegate. Never duplicate — if you find duplicated logic after any future wave, that's a regression, fix it immediately.

8. **Railway prod migration pattern**: Node one-liner using project's `mysql2/promise` dep pointing at `switchback.proxy.rlwy.net:48486` with password from `railway variables --service MySQL --kv`.
