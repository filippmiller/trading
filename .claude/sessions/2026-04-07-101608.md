## Context

User asked to verify the trading app's worker correctly fetches ticker data and stores it. The "worker" is the surveillance sync endpoint (`GET /api/surveillance/sync`) which:
1. Syncs intraday prices for active reversal entries
2. Auto-enrolls new trending stocks (2+ consecutive day movers)

## What Was Done

### Phase 1: Environment Setup
- No MySQL or `.env.local` existed locally. Spun up Docker MySQL container on port 3319.
- Created `.env.local` with `DATABASE_URL=mysql://root:trading123@localhost:3319/trading`
- Started Next.js dev server on port 3013 (avoiding port conflicts per global rules)
- Triggered schema creation via `/api/reversal` endpoint (ensureSchema call)

### Phase 2: Diagnosis
- First sync returned SUCCESS but enrolled 0 entries
- Movers API returned `{"gainers": [], "losers": []}`
- Direct Yahoo Finance screener API test via curl returned valid data (SLNO +32.3%, CAR +11.6%)
- Node.js fetch test also worked — 25 quotes returned
- **Root cause found**: Stooq API blocks automated requests, returning an error message instead of CSV data. `enhanceWithTrend` catches the error silently, returns movers without `consecutiveDays` field (undefined → 0), and the `>= 2` filter in `autoEnrollTrenders` removes everything.
- Secondary issue: sync route was missing `ensureSchema()` call — would crash on fresh database.

### Phase 3: Fix — Yahoo Finance Fallback
- Files: `src/lib/data.ts`
- Changes: Modified `fetchDailyBars()` to try Stooq first, then fall back to Yahoo Finance chart API (`/v8/finance/chart/{symbol}?range=1mo&interval=1d`) when Stooq fails or returns no data.
- Reasoning: Yahoo's chart endpoint provides identical OHLCV daily data. Keeping Stooq as primary preserves backward compatibility for when it works.

### Phase 4: Fix — Missing ensureSchema
- Files: `src/app/api/surveillance/sync/route.ts`
- Changes: Added `import { ensureSchema }` and `await ensureSchema()` at start of GET handler.
- Reasoning: Every other API route calls ensureSchema but sync didn't — would fail on fresh DB.

### Phase 5: Verification
- Re-ran sync: 200 OK in 42s
- Movers API: 10 gainers + 10 losers with trend data (SLNO 4-day streak, CAR 4-day streak, etc.)
- 2 entries enrolled: PAYP (3-day loser → LONG) and SEDG (2-day loser → LONG)
- Surveillance logs: 2 SUCCESS entries
- Data refresh test: AAPL 20 daily bars stored correctly
- Verify endpoint: schema_verified=true, logs visible

## Technical Decisions

| Decision | Rationale | Alternatives Considered |
|----------|-----------|------------------------|
| Yahoo Finance fallback (not replacement) for Stooq | Preserves existing behavior when Stooq works; no breaking change for backtest engine which also uses fetchDailyBars | Full Stooq replacement — but Stooq may work for some users/IPs |
| Yahoo chart `range=1mo` | Sufficient for trend analysis (needs ~5 days of history); matches enhanceWithTrend's `.slice(-5)` window | `range=3mo` — more data but slower, unnecessary for trend check |
| ensureSchema in sync route | Sync was the only major route without it; prevents crash on fresh DB | Could rely on other routes being called first — fragile assumption |

## Gotchas & Notes for Future Agents

1. **Stooq is unreliable for automated use** — blocks requests without prior arrangement. Always ensure Yahoo Finance fallback is in place. If Yahoo also starts blocking, consider Alpha Vantage or Polygon.io as alternatives.
2. **The sync silently succeeds with 0 enrollments** when data sources fail. Consider adding a warning/alert when movers are fetched but 0 pass the consecutiveDays filter — this likely indicates a data source issue rather than genuinely no trending stocks.
3. **Today's enrollments (cohort_date = today) won't have measurement data until tomorrow** — d1_morning/midday/close correspond to cohort_date + 1 business day. This is by design.
4. **The sync route was the only major API route missing `ensureSchema()`** — all other routes have it. If adding new routes, follow the pattern.
5. **Port 3319 was used for MySQL** to avoid conflicts (global rule: add 13 on conflict, but 3306 wasn't in use — used 3319 for safety).
